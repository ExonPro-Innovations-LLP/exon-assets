SOLUTION TYPE
-------------
xVio Scan is a composite solution consisting of:

1. External web application (Progressive Web App at scan.xvio.ai)
2. External API service (REST API at api.xvio.ai)
3. Native mobile apps (iOS and Android)
4. AWS serverless backend (Lambda, DynamoDB, S3, API Gateway)

xVio Scan does NOT include any Salesforce Platform code. There is no managed package, no Apex classes, no Lightning Components, no Visualforce pages, and no custom Salesforce objects. The integration with Salesforce is solely through the Connected App OAuth flow and Salesforce REST API.


SOURCE SCANNER (SALESFORCE)
---------------------------
Not applicable. xVio Scan does not contain Salesforce Platform scannable code (no Apex, Lightning, or Visualforce). There is no managed package to scan via the Salesforce Partner Security Portal.


DAST SCANNER
------------
Dynamic Application Security Testing (DAST) scans were performed using OWASP ZAP 2.17.0 (by Checkmarx) against both production-equivalent targets:

  Target 1: https://api.xvio.ai (REST API)
  Target 2: https://scan.xvio.ai (Web Application)

Scan scope includes all authenticated and unauthenticated endpoints, testing for OWASP Top 10 vulnerabilities including injection, broken authentication, sensitive data exposure, XSS, CSRF, and security misconfiguration.

PREVIOUSLY IDENTIFIED FINDINGS — NOW RESOLVED
-----------------------------------------------
The following findings from earlier scans have been remediated:

  api.xvio.ai (resolved):
  - [HIGH] CORS Misconfiguration: API Gateway Gateway Responses were echoing
    arbitrary Origin headers without validation. Fixed by removing CORS from
    Gateway Responses and making the CloudFront Function the single CORS control
    point with origin whitelist validation. Disallowed origins now receive no
    CORS headers. (Commit: fix(CHG-123))

  - [MEDIUM] Proxy Disclosure: CloudFront proxy detection. Resolved — no longer
    flagged after security header improvements.

  scan.xvio.ai (resolved):
  - [MEDIUM] Content Security Policy Header Not Set: CSP header now configured
    via CloudFront ResponseHeadersPolicy, applied to all responses including
    static assets and error pages.

  - [MEDIUM] Missing Anti-clickjacking Header: X-Frame-Options: DENY now set
    via CloudFront ResponseHeadersPolicy.

  - [MEDIUM] Proxy Disclosure: No longer flagged.

  - [MEDIUM] Sub Resource Integrity Attribute Missing: Google Fonts dependency
    removed. Inter font is now self-hosted from the application origin,
    eliminating the need for SRI on external resources.

  - [LOW] Strict-Transport-Security Not Set: HSTS now configured with
    max-age=63072000 (2 years), includeSubDomains, preload.

  - [LOW] X-Content-Type-Options Missing: nosniff header now set.

  - [LOW] Server Version Leak: "Server: AmazonS3" header suppressed via
    CloudFront ResponseHeadersPolicy.

  - [LOW] Permissions Policy Not Set: Permissions-Policy header now configured
    (camera=(), microphone=(), geolocation=(), payment=()).

  - [LOW] Cross-Origin-Embedder-Policy Missing: COEP header now set.

  - [LOW] Cross-Origin-Opener-Policy Missing: COOP header now set.

  - [LOW] Cross-Origin-Resource-Policy Missing: CORP header now set.


JUSTIFICATION FOR REMAINING MEDIUM FINDINGS
---------------------------------------------
All 3 remaining Medium findings are Content Security Policy (CSP) strictness
issues — not missing security controls. A CSP header IS present and enforced.
The findings relate to specific directive values that are required for the
application to function.

1. CSP: script-src 'unsafe-inline' (CANNOT BE REMOVED)

   xVio Scan is a SvelteKit Progressive Web App built with adapter-static for
   deployment as a Capacitor native mobile app. SvelteKit injects an inline
   bootstrap <script> block in the built HTML that initializes the SPA:

     <script>
       { __sveltekit_1sdnkrc = { base: "" };
         const element = document.currentScript.parentElement;
         Promise.all([import("/_app/immutable/entry/start..."), ...]).then(...);
       }
     </script>

   This inline script is generated at build time by the SvelteKit framework.
   Without it, the application cannot start.

   Why alternatives are not feasible:
   - CSP Nonces: Require per-request server-side generation. Static SPA files
     served from S3/CloudFront cannot have dynamic nonces.
   - CSP Hashes: SvelteKit's csp.mode='hash' generates hashes, but the hash
     changes every build (it includes hashed asset filenames). The CSP is set
     in the CloudFront ResponseHeadersPolicy (AWS infrastructure), which cannot
     be updated dynamically per build without a complex CI/CD pipeline that
     modifies AWS infrastructure on every deployment.

   Mitigation: All other CSP directives are restrictive (default-src 'self',
   frame-ancestors 'none', base-uri 'self', form-action 'self'). No external
   scripts are loaded. The inline script is framework-generated, not
   user-controlled.

2. CSP: style-src 'unsafe-inline' (CANNOT BE REMOVED)

   SvelteKit generates style="display: contents" on the %sveltekit.body%
   wrapper element. Additionally, Svelte components use inline style attributes
   for dynamic styling (e.g., CSS custom properties like
   style="--badge-color: {color}").

   These inline styles are compile-time generated by the Svelte compiler and
   are integral to component rendering. Removing them would require a
   fundamental change to the UI framework.

   Mitigation: No external stylesheets are loaded. All styles are either
   self-hosted CSS files or framework-generated inline styles.

3. CSP: Wildcard Directive — img-src and connect-src (PARTIALLY ADDRESSABLE)

   Current policy:
     img-src 'self' https: data:
     connect-src 'self' https://api.xvio.ai https://api-dev.xvio.ai
                  https://api-stage.xvio.ai wss:

   The img-src https: wildcard exists because tenant branding logos and
   document thumbnails are served from the API, and future tenants may configure
   external logo URLs. The connect-src wss: wildcard covers AWS AppSync
   GraphQL WebSocket subscriptions.

   These wildcards can be tightened to explicit hosts in a future update without
   breaking functionality. This is a planned improvement tracked as a follow-up
   to CHG-123.

   Mitigation: All API connections require authentication (JWT tokens). WebSocket
   connections use AWS AppSync with IAM authorization. The wildcards do not
   expose unauthenticated attack surface.


EXISTING SECURITY CONTROLS
---------------------------
The following security controls are in place and documented in the Technical Solution Overview:

1. AWS WAF v2 with OWASP Common Rules, Known Bad Inputs, SQLi Rules, and IP Reputation lists
2. Application-level rate limiting on all authentication endpoints (DynamoDB-based per-IP counters)
3. API Gateway throttling (1000 req/min, 500 burst)
4. Zod schema validation on all Lambda handler inputs
5. XSS prevention via sanitizeText() on all user inputs
6. CSRF protection via OAuth state parameter (32 bytes, crypto.getRandomValues)
7. PKCE (S256) on all OAuth flows
8. CORS with explicit origin allowlist (no wildcard with credentials) — enforced by CloudFront Function with pattern-matching validation
9. Security headers via CloudFront ResponseHeadersPolicy (HSTS, X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, Referrer-Policy, Permissions-Policy, COEP, COOP, CORP)
10. Content Security Policy with restrictive defaults (default-src 'self', frame-ancestors 'none')
11. TLS 1.2+ enforced on all connections
12. S3 Block Public Access on all buckets
13. AES-256-GCM encryption on all OAuth tokens
14. RS256 JWT with asymmetric signing (private key in Secrets Manager)
15. File upload validation (MIME detection + extension check, size limits)
16. Open redirect prevention (redirect URI allowlist)
17. Server header suppression (AmazonS3 version leak removed)
18. Self-hosted fonts (no external font CDN dependencies)
